Welcome!

Welcome and thanks for your interest in Building a Ruby gem! A few logistics out of the way, this course assumes you feel comfortable writing Ruby code and are generally familiar with the structure of a Rails application. If this isn’t the case, no problem, but be warned, there may be unclear waters ahead…

What is a gem?

If you’re not familiar with Ruby gems already, they’re packaged Ruby libraries or applications that allow you to add functionality to your code. Most gems are isolated features that do things like interact with twit, build a web application, or manage background jobs.

Rubygems.org is the accepted community host of Ruby gems. In a modern Rails application, adding a gem to the Gemfile and running bundle install will cause the application to download the gem so that it can be used locally.

Fortunately for Ruby, the open source eco-system is strong and widespread. There’s very little functionality that a normal application needs that can’t be found on Github.

Why build a gem?

As previously mentioned, gems allow you to isolate features that an application might need. Take Twitter for instance, if you wrote 3 different application that need to post to Twitter as a result of some user action, writing that code 3 different times would be tedious, right?

Even writing it once and copying/pasting to each application would be error-prone. So extracting this code in to a gem would be far easier because you can then reference the gem in each of your applications that need to interact with Twitter and be sure you’re getting the same set of methods and functionality in each.

A more complex topic involves breaking a big applications in to “services” (smaller pieces that focus on specific features). You might do this if your application has gotten really large and managing all the pieces is cumbersome, or you have a large team broken in to smaller teams that each focus on isolated parts of the application. This is beyond the scope of this course, but suffice it to say, there’s a few other reasons you might extract code from your application in to a gem.

I wrote about popular Ruby gems in my applications and looking at this list again, it’s amazing to see the benefits I gained from including external code. It’s one of the reasons why building a web application using Rails allows you to iterate so fast.

According to Rubygems.org, since July 2009, there have been 66,783 gems created of which have been downloaded a total of almost 2.4 billion times (wild right?). That’s a lot of downloads and a lot of gems! The open source world is a better place for it and so are your applications. Take some time to simplify your applications and let others take advantage of things that you find valuable in your own code by creating an open source Ruby gem.

Open source or private?

The Ruby gem that we’ll be building as part of this course will be open source. That is, free for all to use however they wish. However, as mentioned above, there are situations where breaking a large applications in to smaller gems is a good idea, in which case, it’s completely reasonable for those smaller gems to be private within your organization. Github makes it really easy to create a code repository that’s private, so that you can keep this code protected.

Extract code

In a previous post, I wrote about 3 ways to get started contributing to open source. The last point suggests extracting code from an existing application so that you can reuse this functionality elsewhere. And that’s what we’ll be doing in this course.

The beauty of removing existing code from your application is:

  1. Your application becomes simpler
  2. Your gem becomes reusable
  3. Other people can benefit from your new gem

So_meta

So what are we building?!?!

I write a lot of smallish Rails web applications. As a responsible web developer (that’s us, right?), I do my best to include good SEO practices in all my publicly facing pages. To my knowledge, there’s no easy way to do this. I’ve tried adding this information to the individual views, but this approach quickly turned in to a mess.

I later stumbled on this gist that caught my eye. I tried it in a few applications and fell in love with its simplicity and organization. And as a result, I extracted this code in to a gem called so_meta.

The “gist” (pun intended!) of the code is that you can take advantage of Rails’ built-in support for internationalization by adding your SEO meta tags to the config/locales/en.yml file. If you’ve not had a need to translate an application to another language, you’ve likely not touched this file before. However, I think you’ll find that there are plenty of uses for it beyond simple language translation.

In Summary

Hopefully you now see why we’d want to build a Ruby gem and some of the benefits we’ll get from doing so. The next post in this course will discuss some of the conventions used within a gem and we’ll finally get to looking at some…drumroll…code! :)
